Amazon ECS는 Amazon CloudWatch와 밀접하게 연결되어 CloudWatch가 제공하는 실시간 메트릭을 기준으로 효과적인 스케일링을 할 수 있습니다. ECS는 1분 간격으로 CloudWatch에 메트릭을 제공합니다. 메트릭이 스케일링 정책에 정해둔 수치를 초과하거나 미달할 때 CloudWatch Alarm이 발생하고, 이를 바탕으로 태스크 수 조절이 수행됩니다. 이후 Application Auto Scaling이 `desired count`를 변경하고, 변경된 `desired count`에 맞춰 ECS가 태스크 수를 조절합니다. 

Amazon ECS는 4개의 스케일링 전략(Scaling Strategies)을 제공합니다. 

## Target Tracking Scaling

Target Tracking Scaling은 목표 메트릭 하나를 정해두고 그 메트릭을 기준으로 태스크 수를 조절하는 스케일링 전략입니다. 

예를 들면, CPU 사용률을 메트릭으로 정하고 사용률이 60%에 가깝게 유지되도록 태스크 수를 조절할 수 있습니다.

각 태스크에는 지정된 CPU와 메모리 자원이 할당되고, 이는 Task Definition에서 정할 수 있습니다. 태스크가 추가될 때마다 요청이 추가된 태스크로 분산되기 때문에 CPU 사용률이 줄어들 수 있습니다.

이 스케일링 전략은 섬세한 스케일링이 요구되지 않는 서비스에서 주로 사용되며, 사용자의 직접적인 개입이 자주 필요하지 않기 때문에 간편한 운영의 이점이 있습니다.

## Step Scaling

Step Scaling은 Target Tracking Scaling보다 더 섬세한 스케일링 작업이 가능합니다. 여러 개의 메트릭을 기준으로 태스크 수를 조절하는 스케일링 정책을 만들 수 있습니다. 이때 메트릭은 여러 종류의 메트릭을 기준으로 하는 것이 가능합니다. 

예를 들면, CPU 사용률 기준으로 70% 이상이면 태스크를 1개 추가하고, 90% 이상이면 태스크를 3개 추가하며, 메모리 사용률 기준으로 80% 이상이면 태스크를 1개 추가하는 스케일링 정책을 만들 수 있습니다. 

또한 Step Scaling은 Breach Evaluation Periods를 사용자가 직접 설정할 수 있습니다. 이 설정은 스케일링을 유발시키는 임계값(threshold)에 도달했을 때 바로 스케일링을 발생시키지 않고, 특정 시간 내에 특정 횟수만큼 임계값에 도달했을 때 스케일링을 유발시키는 것을 의미합니다. 이 설정을 통해 더 확실한 상황에서 스케일링할 수 있습니다.

## Scheduled Scaling

Scheduled Scaling은 정해진 시간대에 스케일링을 하는 전략입니다. 예를 들면 은행 애플리케이션 같은 경우 정오부터 1시까지 이용률이 상승하는 경향이 있는데, 이 시간대에 태스크 수를 자동으로 늘리는 것입니다. 이는 예측 가능한 트래픽에 효과적인 스케일링 전략입니다.


## Predictive Scaling

Predictive Scaling은 기계 학습을 이용해 트래픽을 예측해 태스크 수를 조절하는 방법입니다. 충분한 데이터가 필요하기 때문에 장기적으로 운영된 애플리케이션에서 사용될 수 있습니다.
